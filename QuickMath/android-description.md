## Potential Issues & Risks

### 1. Infinite Loading Screen on Android (FIXED)
**Severity: High ‚ÄĒ FIXED**

The `index.html` contained a CSS animation loader inside `<div id="app">`, but no code was hiding it when Blazor finished loading. On Android, this caused the loader to persist indefinitely.

**Fix applied:** Added JavaScript to `index.html` that listens for `Blazor.addEventListener('initialized', ...)` and hides the `.loader` element once Blazor is ready.

```javascript
Blazor.addEventListener('initialized', function () {
    var loader = document.querySelector('.loader');
    if (loader) {
        loader.style.display = 'none';
    }
});
```

### 2. Missing CSS Variables in MAUI wwwroot
**Severity: Medium ‚ÄĒ FIXED**

The `Home.razor` page uses CSS variables (`--bg`, `--accent`, `--text`, `--muted`) that were not defined in the MAUI `wwwroot/app.css`. This caused unstyled content on Android.

**Fix applied:** Added `:root` CSS variables to `QuickMath/QuickMath/wwwroot/app.css`.

### 3. Package Name Mismatch
**Severity: Low**

The `.csproj` declares:
```xml
<ApplicationId>com.companyname.quickmath</ApplicationId>
```

But `AndroidManifest.xml` declares:
```xml
package="com.companyname.quickmath"
```

The manifest `package` attribute takes precedence. This inconsistency can cause confusion during publishing, Firebase project linking, or Play Store registration. **Recommendation:** Align both to the same value.

### 6. WebView Performance & Compatibility
**Severity: Medium**

- `BlazorWebView` on Android uses the system WebView (Chromium-based)
- The min SDK is API 30 (Android 11), which ships with a modern WebView that supports current CSS/JS features well
- **Note:** With API 30+, WebView compatibility is significantly better than with the previously-documented API 24

### 7. Color Theme Inconsistency
**Severity: Low-Medium**

- `colors.xml` uses `#512BD4` (default .NET purple)
- App CSS theme uses `#8e24ff` (lighter purple)
- The splash screen and status bar during launch will show a different shade than the running app
- **Recommendation:** Update `colors.xml` to match the app's accent color (`#8e24ff` / `#6a1bcc` for dark variant)

---

### 8. No ProGuard/R8 Configuration
**Severity: Medium (for release builds)**

- No `proguard.cfg` or R8 rules file exists
- BlazorWebView and AOT-compiled .NET code may get incorrectly stripped in Release builds
- When building for release, add appropriate keep rules for MAUI and Blazor types

### 9. No Network Security Config
**Severity: Medium (if using local dev server)**

- No `network_security_config.xml` defined
- If the app needs to communicate with a local development server (HTTP), Android 9+ (API 28+) blocks cleartext traffic by default
- The `INTERNET` permission is declared, but cleartext is not explicitly allowed
- **To fix:** Add a `network_security_config.xml` to allow cleartext for development domains, or ensure all traffic uses HTTPS

---

### 10. No Hardware Acceleration Explicitly Set
**Severity: Low**

- `BlazorWebView` benefits from hardware acceleration for smooth rendering
- The manifest doesn't explicitly set `android:hardwareAccelerated="true"` on the `<application>` tag
- It's enabled by default on API 14+, but being explicit avoids edge cases

---

### 11. ConfigurationChanges May Mask Issues
**Severity: Low**

The `ConfigurationChanges` flags prevent the activity from being recreated on rotation, but:
- The BlazorWebView may not properly handle all configuration changes without a refresh
- Some resources (like CSS media queries for orientation) won't re-evaluate until the WebView is notified
- **Recommendation:** Test rotation thoroughly; consider removing `ConfigChanges.UiMode` if dark/light mode switching needs to trigger a proper theme reload

---

**Severity: Medium (for distribution)**

The `.csproj` doesn't specify `<RuntimeIdentifiers>` for Android:
- Default will build for the host architecture only during debug
- For release/APK distribution, you'll need to specify `android-arm;android-arm64` at minimum
- Without this, the published APK may not work on all devices

## Build & Publish Commands

### Debug on Android (emulator)
```bash
dotnet build -t:Run -f net10.0-android
```

### Debug on connected device
```bash
dotnet build -t:Run -f net10.0-android --device
```

### Publish APK (debug)
```bash
dotnet publish -f net10.0-android -c Debug
```

### Publish AAB (release, for Play Store)
```bash
dotnet publish -f net10.0-android -c Release
```

Output location: `bin/Release/net10.0-android/publish/`
